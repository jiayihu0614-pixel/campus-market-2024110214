# Day3 Evidence：Mock 数据建模与列表渲染

## 1. 今日完成内容
  今天在已完成页面骨架、路由导航和公共布局的基础上，我继续推进校园轻集市Day3的数据驱动改造工作。先完成axios与json-server依赖安装，在项目根目录新建db.json配置文件，同时配置pnpm mock脚本用于快速启动本地mock服务。按照业务模块对接口请求代码进行拆分封装，开发通用信息卡片组件与空状态提示组件，将二手交易、失物招领、拼单搭子、跑腿委托四个原本写死静态内容的页面，改造为通过后端接口拉取数据并循环渲染列表的动态页面。

## 2. Mock 数据设计说明
我结合校园真实使用场景梳理出四类核心业务数据：二手交易模块重点记录商品标题、售价、所属分类、物品成色、线下交易地点；失物招领区分寻物、招领两类场景，记录遗失或捡拾物品名称、事件发生时间、事发地点以及发布人联系方式；拼单搭子模块记录活动类型、招募目标人数、当前参与人数、报名截止时间；跑腿委托包含任务类型、取件地址、送达地址、任务酬劳、约定完成时间。四类业务各设计5条模拟数据，素材选用教材、有线耳机、校园卡、奶茶拼单、代取快递等贴合校园日常的真实场景。

## 3. 我的设计
本次Day3的数据模型设计优先适配前端列表页展示需求，没有直接按照完整后端数据库标准做冗余设计。统一提取标题、描述、地理位置、发布时间、发布用户、业务状态这类多页面共用字段，在此基础上为四类业务分别拓展专属属性字段：二手商品新增price售价、condition成色字段；拼单模块设计currentCount当前参与人数、targetCount目标招募人数字段；跑腿任务单独设置pickupLocation取件地址、deliveryLocation送达地址、reward酬劳字段。该设计既能清晰区分不同业务的数据特征，又可以避免数据结构过于繁杂冗余，降低后续前端字段绑定与调试的工作量。

## 4. AI 设计
  本次开发借助codex完成项目结构梳理、依赖安装命令编写、db.json初始模板生成、axios接口分层代码编写以及Vue列表渲染基础代码开发。借助工具可以快速统一全局字段命名规范，一次性补齐四类业务页面所需的展示字段，大幅缩减重复编码耗时。但工具默认生成的模板经常会额外拓展用户表、订单表、评论表、收藏表、权限控制等关联数据表，这类拓展内容不在本次Day3开发范围内，不仅会提升代码理解难度，还会增加调试出错概率，因此并未全部采纳工具生成的完整数据结构方案。

## 5. 人工审查与调整
  我对工具生成的初始代码和数据结构做了逐段人工审核精简，仅保留作业要求的四张业务数据集以及页面展示必需的核心字段，舍弃了登录身份令牌、订单编号、评论统计、收藏关联、内容审核权限等超出当前开发阶段的冗余设计。逐一完成页面数据字段绑定校验：将跑腿任务的reward酬劳字段渲染在卡片价格展示区域，取件、送达两个地址做合并展示；拼单模块的人数信息整合至卡片描述区域；依据布尔类型字段区分渲染寻物、招领标签。同时校验db.json语法格式、项目文件相对路径、原有路由配置是否保持一致，确认Day1、Day2阶段开发的原有页面、组件文件没有出现误删除、代码覆盖等问题。

## 6. JSON Server 接口模拟
  项目依托根目录下的db.json文件，通过JSON Server搭建本地模拟接口服务，服务固定占用3001端口，启动后可通过/trades、/lostFounds、/groupBuys、/errands四个接口地址分别请求四类业务数据。前端Vite开发服务与本地Mock接口服务需要分开启动运行，若仅启动前端项目未开启Mock服务，页面会因接口请求失败触发异常提示，提醒开发者检查模拟接口服务运行状态。本阶段使用本地Mock接口的核心目的是练习前后端数据交互流程，并非开发正式后端业务系统。

## 7. Axios 请求封装
  在src/api/http.ts文件中初始化Axios基础请求实例，统一配置接口基础请求地址与请求超时时间；按照四类业务拆分独立接口文件，分别编写trade.ts、lostFound.ts、groupBuy.ts、errand.ts接口请求模块。各个业务页面仅需要引入对应模块下的数据查询方法即可发起网络请求，无需在每个页面重复编写接口地址，本次仅做基础请求封装，没有接入身份校验、复杂请求响应拦截、登录强制跳转等拓展功能。

## 8. 页面列表渲染
  四个业务页面均采用ref声明响应式数组存储接口返回列表数据，在onMounted生命周期钩子函数内调用对应接口拉取后端数据，通过v-for指令循环遍历数据渲染通用ItemCard.vue组件。针对接口请求失败、暂无业务数据两种场景，引入EmptyState.vue空状态组件展示友好提示文案。页面内容不再通过模板硬编码实现，完全由接口返回数据驱动渲染，后续仅需要修改db.json内模拟数据，或是直接替换线上正式接口地址，不需要改动页面布局与渲染逻辑。

## 9. 遇到的问题
  安装依赖阶段发现最新测试版本的json-server移除了教程中用到的--watch监听参数，为保证项目能够正常监听db.json文件修改并自动刷新接口数据，我降级安装了支持该参数的稳定版本；同时注意到db.json文件必须严格遵循标准JSON语法规范，不能存在注释、多余逗号等非法格式；前端开发服务和Mock接口服务必须同时运行，一旦其中一个服务中断就会出现接口请求失败、页面无法渲染列表的问题。

## 10. 后续计划

后续准备继续在当前 Mock 数据和 API 分层的基础上完善必要的加载提示、基础筛选与发布表单，并继续检查组件复用和字段映射是否合理。Day3 暂时不加入登录鉴权、订单、评论、收藏和复杂消息功能，先保证四类核心业务的数据请求与列表展示稳定，为后续实验保留清晰、可扩展的代码结构。
