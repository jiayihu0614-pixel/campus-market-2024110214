# Day6 Evidence - Mock注册登录与交互完善

## 1.今日完成内容

今天我在前几天已经完成页面骨架、Mock数据、发布表单和Pinia状态管理的基础上，继续完善了校园轻集市的用户操作流程。主要完成了Mock注册登录、用户状态持久化、退出登录、四类信息的关键词搜索、详情页、发布图片、私聊消息、拼单报名和个人中心最近动态。项目现在不只能展示列表，还能完成“浏览—查看详情—联系发布者—报名或发布—个人中心查看记录”的基础闭环。

## 2.Mock注册登录

我在db.json中使用users数据集保存Mock用户，注册时需要填写用户名、密码、昵称、学院、年级和校区等信息。页面会检查必填项、密码长度、两次密码是否一致，也会检查用户名是否已存在。登录成功后，顶部导航显示当前用户和退出按钮；未登录时，发布页、消息中心和个人中心会提示先登录。这里只是JSON Server的Mock登录，没有使用JWT、Token和真实权限系统。

## 3.状态持久化与Store设计

userStore保存isLoggedIn和currentUser，登录后将当前用户资料写入localStorage，刷新页面后再由App.vue恢复，退出时同时清理Store和本地记录。favoriteStore负责跨页面共享收藏状态，activityStore负责保存当前用户报名参加的活动。我没有把搜索关键字、加载状态和表单校验错误放入Pinia，因为这些内容只在当前页面使用。这次让我更清楚地理解了状态管理的边界：需要多页面同步的数据才适合放入Store。

## 4.搜索与页面反馈

二手交易、失物招领、拼单搭子和跑腿委托四个页面都加入了关键词搜索。不同页面会按标题、类型、物品名称、地点和描述等字段过滤。页面请求数据时显示LoadingState，请求失败时显示ErrorState和重试按钮，无搜索结果时显示EmptyState。这些交互优化让我意识到，页面不能只考虑“有数据”的情况，还要给用户清楚的等待、失败和空数据反馈。

## 5.信息详情与图片发布

首页保留简单信息，点击后可进入统一详情页。四类列表中的信息标题也能进入对应详情。详情页根据业务类型展示价格、成色、物品名称、参与人数、酬劳、截止时间等不同字段，底部还有同类相关推荐。如果该信息不是当前用户发布的，页面会显示“联系发布者”；如果是自己发布的，则显示本人发布提示。

发布页增加了非必填图片上传，可以预览、重新选择和移除图片，图片不能超过1.5MB。没有图片时仍然可正常发布，有图片时详情页会在主内容顶部展示。目前db.json中保留了一条由“计算机科学学院 周同学”发布的带图二手教材信息，可以从列表进入详情查看。

## 6.私聊消息与页面联动

消息中心从原来的静态提示改成了左侧会话列表和右侧聊天窗口，预置了教材、失物和跑腿三类校园对话。当前用户可以切换联系人、查看未读数和发送回复，聊天内容按账号保存在localStorage。从详情页点击联系发布者后，消息中心会自动建立与该同学的会话，并显示正在咨询的信息标题。这里没有使用WebSocket或真实消息推送，仍符合前端Mock实训的范围。

## 7.拼单报名与最近动态

拼单详情页增加了“报名参加”入口。报名页显示当前人数、目标人数、集合地点、截止时间和发布者。报名成功后，通过PATCH请求将currentCount加1；如果报名后达到targetCount，同时将status改为closed。人数已满、活动已关闭、报名自己发布的活动或同一账号重复报名时，页面都会阻止操作。

报名成功后，activityStore会保存活动名称、类型、地点、截止时间和报名时间。个人中心增加了“已报名”数量和“最近动态”时间线，按报名时间从新到旧排列，并可以返回活动详情。这一部分让我看到了Pinia对多页面数据同步的实际作用。

## 8.AI协作记录

今天我使用AI Coding工具协助检查当前项目结构，并辅助生成了登录注册、详情页、消息页、报名页和Activity Store的初版代码。AI在快速搭建跨页面流程方面效率比较高，但生成的内容不能直接当成最终结果。我需要根据实训文档判断功能是否超出范围，也需要检查字段名、路由、当前用户判断和JSON Server提交路径。

## 9.人工检查与修改

我对AI生成的内容进行了人工调整。我删除了网站中“功能验证”、“实训同学”等不真实的展示文案，将发布人和信息内容改成符合校园场景的文字。我还修正了AppHeader.vue中误输入字符导致的CSS构建错误，检查了不是本人发布时才显示联系入口，并为报名功能增加了满员、已关闭、本人发布和重复报名检查。

## 10.真实测试记录

测试时我同时启动了pnpm run mock和pnpm run dev，检查了首页、四类列表、详情页、发布页、消息中心、报名页和个人中心的访问状态。带图二手教材的图片和详情地址均返回200。拼单报名流程使用临时Mock数据测试，成功将人数从1/2更新为2/2，并将状态从open改为closed；测试后已删除临时记录，没有把“测试”文案留在网站中。最后执行pnpm run lint和pnpm run build，ESLint、TypeScript检查和Vite构建均通过。

## 11.今日反思

通过今天的完善，我发现一个看起来简单的校园信息平台，需要考虑用户是否登录、数据是否加载成功、信息是否本人发布、报名是否满员以及多页面如何同步等问题。交互优化不只是让页面更好看，更重要的是让用户在每一步都知道当前状态和下一步操作。后续如果继续开发，可以再将目前保存在localStorage的私聊和报名记录迁移到真实后端，但这部分不属于当前Day6实训范围。
