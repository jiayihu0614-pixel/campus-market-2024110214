# Day6 Evidence - Mock注册登录与交互完善

## 1.今日完成内容

今天我在Day5已经完成Pinia状态管理、收藏和个人中心的基础上，继续完成了Day6的Mock注册登录、状态持久化和交互优化。我为JSON Server增加了users数据集，补充了用户API、登录页、注册页和对应路由。导航栏现在能根据用户状态显示登录注册入口，或显示当前用户、收藏数量和退出按钮。发布页和个人中心也增加了游客状态提示，避免未登录时把发布人错写为游客。

## 2.Mock注册登录设计

注册页收集用户名、密码、昵称、学院、年级、校区和简介，校验必填项、密码长度、两次密码是否一致，并在提交前检查用户名是否重复。登录页从users接口查找匹配账号，匹配成功后只把必要的用户资料交给Store，不把密码保存到Pinia或localStorage。这只是实训所需的Mock流程，没有加入JWT、Token、权限路由或真实后端认证。

## 3.用户状态与持久化

userStore管理isLoggedIn和currentUser，登录时将当前用户写入localStorage，项目重新打开时由App.vue调用restoreLogin恢复状态，退出时同时清除Store和localStorage。我理解的状态边界是：当前用户和收藏需要被多个页面共享，适合放入Pinia；搜索关键字、加载状态、错误状态和表单临时内容只属于当前页面，仍保留在组件内部。

## 4.交互优化

二手交易页增加了SearchBar、LoadingState和ErrorState组件。搜索可以按标题、分类、地点和描述即时过滤；请求中显示加载提示；Mock服务不可用时显示错误说明和“重新加载”按钮；搜索无结果时继续复用EmptyState。这些修改只补齐用户反馈，没有改动Day3的数据字段、Day4的POST接口和Day5的收藏逻辑。

## 5.AI协作与人工审查

本次我让AI Coding工具帮助检查现有项目、生成用户API、登录注册页、Store持久化和通用反馈组件的初版代码。我重点核对了users字段、路由地址、发布人来源和退出后的状态。人工审查时，我保留了简单的JSON Server账号查询，没有接受真实鉴权、密码加密、权限系统等超出Day6范围的设计。我还检查了登录成功时的字段映射，确认密码不会进入持久化状态。

## 6.真实测试记录

我同时启动了pnpm run mock和pnpm run dev，确认users、trades、lostFounds、groupBuys和errands五类接口可以访问。测试时真实新增了“Day6实训同学”账号，并新增一条“Day6功能走查：出闲置前端开发教材”二手信息，发布人为该用户。接口返回的用户id为2，二手信息id为9，说明数据已写入db.json。我还检查了首页、四类业务页、发布页、消息页、个人中心、登录和注册页，所有路由均正常返回。最后执行pnpm run build，类型检查和Vite构建均通过。

## 7.遇到的问题

登录初版代码为了去掉password使用了一个未使用的解构变量，ESLint检查时会报错。我将它改成显式映射CurrentUser需要的字段，既通过了检查，也更直观地说明了哪些资料可以保存。此外，未登录时displayName会返回“游客”，如果不额外检查，发布数据就可能出现错误发布人。现在提交前会先判断isLoggedIn，并带上redirect返回发布页。

后续可以继续做人工浏览器走查，重点检查刷新后登录状态、收藏按钮与个人中心的联动，以及关闭Mock服务后的错误提示。本项目仍然是前端实训用Mock系统，不应将现有明文密码和本地状态方案直接用于真实产品。
