# Day2 Evidence - 页面骨架与路由导航

## 1. 今日完成内容

今天我在 Day1 项目能够正常运行的基础上，完成了“校园轻集市”PC 端 Web App 的页面骨架搭建。根据项目的核心业务，我创建并整理了首页、二手交易、失物招领、拼单搭子、跑腿委托、发布信息、消息中心和个人中心，共 8 个页面。每个页面都有明确的中文标题、页面说明和与业务相关的基础内容，但暂时没有加入真实接口、数据库和复杂交互，符合 Day2 先搭结构、再逐步开发功能的要求。

今天还完成了公共布局拆分。我创建了 `AppLayout.vue`、`AppHeader.vue` 和 `AppNav.vue` 三个通用组件。`AppLayout` 负责整体页面容器，`AppHeader` 负责品牌与顶部区域，`AppNav` 负责导航入口和当前页面高亮。`App.vue` 只加载统一布局，各业务页面通过 `RouterView` 显示。除此之外，我新增了 `src/assets/main.css`，统一设置页面背景、标题、卡片和基础响应式样式。

## 3. AI 协作记录

今天使用的 AI 工具是 Codex。我的核心提示词是：根据 Day2 实训要求，在当前种子代码基础上创建 8 个业务页面，配置 Vue Router，拆分公共布局组件，并且不要修改 Day1 Evidence 和 `docs/ai` 中的内容。

AI 辅助生成了页面骨架、路由配置、公共组件和全局样式的初始代码，也协助运行了 TypeScript 检查、Lint、Vite 构建和路由访问测试。我没有直接把 AI 输出当成最终结果，而是按照任务要求进行了人工检查和修改。我逐项核对了页面名称、文件名、路由路径和导航文字是否对应，确认没有出现 `/page1`、`/test` 等无意义路径；检查了每个页面的说明是否符合校园二手交易、失物招领、拼单搭子和跑腿委托等场景；同时确认公共布局组件和页面组件的职责没有混在一起。我还要求删除与本次 Day2 任务不一致的复杂原型方案，重新按简单、清晰、可扩展的页面框架实现。

## 4. 遇到的问题与解决方法

代码完成后，我第一次逐个访问路由时，页面显示无法连接。开始时我怀疑是路由路径或组件导入错误，但 TypeScript 检查和生产构建都已经通过。继续检查后发现，真正原因是原来的 Vite 开发服务器已经停止，并不是路由代码出错。重新启动开发服务器后，我依次访问 `/`、`/trade`、`/lost-found`、`/group-buy`、`/errand`、`/publish`、`/message` 和 `/user`，8 个路径都能够正常返回和显示。

另外，种子首页原来存在中文乱码。搭建新页面时，我重新填写了正常的中文标题和说明，并检查了文件编码。最后运行 `vue-tsc`、ESLint、Oxlint 和 Vite 构建，确认代码没有明显的类型错误和编译错误。这个过程让我认识到，出现页面打不开的问题时，应该分别检查开发服务器、路由、组件路径和浏览器访问地址，而不是盲目修改代码。

## 5. 今日反思

通过今天的任务，我理解了页面骨架、路由导航和公共布局之间的关系。页面骨架先确定每个业务页面负责什么内容，路由导航建立页面之间的访问关系，公共布局则保证所有页面共享统一的顶部区域、导航和内容容器。先把这些基础结构搭清楚，后续增加商品列表、发布表单、聊天消息和个人数据时，就可以在对应页面中继续开发，而不需要反复调整整个项目。我也认识到 AI 可以帮助提高创建文件和排查问题的效率，但页面是否符合实训要求、组件边界是否合理、路由是否真正可以访问，仍然需要我自己逐项检查和判断。
